1. What is Docker? Beginner

Docker is a platform for building, shipping, and running applications inside containers. A container packages your application together with everything it needs to run — code, runtime, system libraries, and settings — into a single, isolated, portable unit.

The problem it solves

The classic developer complaint: "It works on my machine!" Applications behave differently across environments because of mismatched OS versions, library versions, missing dependencies, and configuration drift. Docker eliminates this by bundling the app and its dependencies into an image that runs identically everywhere — your laptop, a teammate's machine, CI, staging, and production.

💡 Containers vs Virtual Machines A VM virtualizes hardware and runs a full guest OS (heavy, GBs, slow to boot). A container virtualizes the OS — all containers share the host kernel but are isolated from each other (lightweight, MBs, boots in milliseconds). You can run dozens of containers where you'd fit only a few VMs.

2. Docker Architecture Beginner

Docker uses a client–server architecture.

┌─────────────┐ REST API ┌──────────────────────────────────┐ │ Docker │ ───────────────▶ │ Docker Daemon (dockerd) │ │ Client │ docker build │ ┌────────────┐ ┌─────────────┐ │ │ (docker CLI)│ docker pull │ │ Images │ │ Containers │ │ └─────────────┘ docker run │ └────────────┘ └─────────────┘ │ │ ┌────────────┐ ┌─────────────┐ │ │ │ Volumes │ │ Networks │ │ │ └────────────┘ └─────────────┘ │ │ containerd → runc (OCI runtime)│ └─────────────────┬────────────────┘ │ pull / push ┌───────▼────────┐ │ Registry │ │ (Docker Hub, │ │ ECR, GHCR...) │ └────────────────┘
ComponentRole
Docker ClientThe docker CLI you type commands into. Sends them to the daemon via REST API.
Docker Daemon (dockerd)The background service that builds, runs, and manages images, containers, networks, and volumes.
containerd / runcLower-level runtimes the daemon uses to actually create and run containers (OCI standard).
ImagesRead-only templates used to create containers.
ContainersRunning (or stopped) instances of images.
RegistryStorage & distribution for images (Docker Hub, GHCR, ECR, GCR, private registries).
💡 Image vs Container — the key mental model An image is like a class; a container is like an instance of that class. One image can spawn many containers. Images are immutable and read-only; containers add a thin writable layer on top.

# Linux — official convenience script (great for dev/test)
curl -fsSL https://get.docker.com -o get-docker.sh

# Run docker without sudo (log out/in afterwards)
sudo usermod -aG docker $USER

# Verify the install
docker version            # client + server versions
docker info               # system-wide details
docker run hello-world     # the classic first test
⚠️ The docker group = root Adding a user to the docker group grants effectively root-level access on the host. Only do this for trusted users. For tighter security, consider rootless Docker or Podman.

4. Your First Container & Core CLI Beginner

The fundamental command is docker run, which pulls an image (if needed) and starts a container from it.

bash
# Run nginx in the background, map host port 8080 -> container 80
docker run -d -p 8080:80 --name web nginx:1.27

# Run an interactive shell in an Ubuntu container (removed on exit)
docker run -it --rm ubuntu:24.04 bash

# Run a one-off command
docker run --rm alpine echo "hello from a container"

Common docker run flags

FlagMeaning
-dDetached — run in the background.
-itInteractive + TTY — for shells.
--rmAuto-remove the container when it stops.
-p host:containerPublish/map a port.
--nameGive the container a friendly name.
-e KEY=valSet an environment variable.
-v src:dstMount a volume / bind mount.
--networkAttach to a specific network.
--restartRestart policy (no, on-failure, always, unless-stopped).

Managing containers

bash
docker ps                     # running containers
docker ps -a                  # all (incl. stopped)
docker stop web               # graceful stop (SIGTERM then SIGKILL)
docker start web              # start a stopped container
docker restart web
docker rm web                 # remove (must be stopped, or -f)
docker rm -f web              # force remove a running container
docker logs -f web            # stream logs
docker exec -it web sh        # shell into a running container
docker inspect web            # full JSON details
docker stats                  # live CPU/mem/IO usage
docker top web                # processes inside the container
docker cp web:/etc/nginx/nginx.conf ./   # copy files out

Managing images

bash
docker images                 # list local images
docker pull redis:7           # download an image
docker rmi redis:7            # remove an image
docker tag myapp:1.0 myrepo/myapp:1.0   # retag
docker history nginx:1.27     # layers of an image
docker image inspect nginx:1.27

Understanding image tags

An image reference looks like registry/repository:tag, e.g. ghcr.io/acme/api:1.4.2. If you omit the registry it defaults to Docker Hub; if you omit the tag it defaults to :latest.

⚠️ Never rely on :latest in production :latest is just a tag, not "the newest" in any guaranteed sense — and it changes over time, breaking reproducibility. Always pin explicit versions (ideally by digest image@sha256:...) for production deployments.

5. Building Images: The Dockerfile Intermediate

A Dockerfile is a text recipe of instructions to build an image. Each instruction creates a layer.

dockerfile — a Node.js app
FROM node:20-alpine                  # base image
WORKDIR /app                          # working dir for following cmds

# Copy manifests first to leverage layer caching
COPY package*.json ./
RUN npm ci --omit=dev                  # install deps

COPY . .                               # copy the rest of the source

ENV NODE_ENV=production
EXPOSE 3000                            # documents the port
USER node                              # drop root

HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "server.js"]            # default process
bash
# Build (tag with -t). The "." is the build context.
docker build -t myapp:1.0 .

# Build with a specific Dockerfile / build args
docker build -f Dockerfile.prod --build-arg VERSION=1.0 -t myapp:1.0 .

# Run your freshly built image
docker run -d -p 3000:3000 --name api myapp:1.0

Dockerfile instructions reference

InstructionPurpose
FROMBase image to build on. (Required, first line.)
WORKDIRSet the working directory for subsequent instructions.
COPYCopy files from build context into the image.
ADDLike COPY but also handles URLs & auto-extracts tarballs (prefer COPY).
RUNExecute a command at build time (creates a layer).
CMDDefault command at run time (can be overridden).
ENTRYPOINTThe fixed executable; CMD becomes its default args.
ENVSet environment variables (persist into containers).
ARGBuild-time variable (not present at run time).
EXPOSEDocument which port the app listens on (informational).
VOLUMEDeclare a mount point for persistent/external data.
USERSet the user to run as (use a non-root user!).
HEALTHCHECKCommand Docker uses to test container health.
LABELAdd metadata (maintainer, version, source).

CMD vs ENTRYPOINT

  • CMD ["nginx", "-g", "daemon off;"] — easily overridden by docker run image <other-cmd>.
  • ENTRYPOINT ["python", "app.py"] — fixed; args after the image name are appended.
  • Combine them: ENTRYPOINT ["python"] + CMD ["app.py"] → defaults to python app.py but you can swap the script.
✅ Use exec form, not shell form Prefer the JSON array (exec) form CMD ["node","server.js"] over the string (shell) form CMD node server.js. Exec form makes your process PID 1 so it receives signals (SIGTERM) directly — essential for graceful shutdown.

6. Layers, Caching & the Build Context Intermediate

Images are made of stacked, read-only layers. Each RUN, COPY, and ADD creates a new layer. Docker caches layers and reuses them on rebuild — as long as the instruction and its inputs haven't changed. Once a layer changes, every layer after it is rebuilt.

✅ Order instructions from least to most frequently changing Copy dependency manifests (package.json, requirements.txt, go.mod) and install before copying your source code. Your source changes constantly, but dependencies rarely — so the expensive install layer stays cached across most builds.

The .dockerignore file

The "build context" is everything in the directory you pass to docker build. A .dockerignore keeps junk out — shrinking the context, speeding builds, and avoiding leaking secrets.

.dockerignore
.git
node_modules
npm-debug.log
Dockerfile*
.dockerignore
.env
*.md
dist
coverage
.vscode

Combine RUN steps to reduce layers

dockerfile
# BAD — 3 layers, cache leftover apt lists bloat the image
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# GOOD — single layer, cleaned up in the same step
RUN apt-get update \
 && apt-get install -y --no-install-recommends curl \
 && rm -rf /var/lib/apt/lists/*

7. Multi-Stage Builds Advanced

One of the most important techniques for production. Use a heavy "builder" stage to compile your app, then copy only the resulting artifacts into a tiny final image — leaving behind compilers, dev dependencies, and source code.

dockerfile — multi-stage Go build
# ---- Stage 1: build ----
FROM golang:1.22 AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app/server ./cmd/server

# ---- Stage 2: minimal runtime ----
FROM gcr.io/distroless/static-debian12
COPY --from=builder /app/server /server
USER nonroot:nonroot
EXPOSE 8080
ENTRYPOINT ["/server"]
✅ The payoff A Go app this way can shrink from ~900 MB (full golang image) to ~10–20 MB. Smaller images = faster pulls, smaller attack surface, lower storage/bandwidth cost. Use distroless, alpine, or scratch for final stages.

Choosing a base image

BaseSizeNotes
scratch0 MBEmpty. For fully static binaries (Go, Rust). No shell.
distroless~2–20 MBJust your app + runtime, no shell/package manager. Very secure.
alpine~5 MBTiny, has a shell & apk. Watch out for musl libc quirks.
debian-slim~30 MBGlibc, broad compatibility, small.
ubuntu~75 MBFamiliar, full-featured, larger.

8. Data Persistence: Volumes & Mounts Intermediate

Containers are ephemeral — when one is removed, its writable layer (and any data written inside) is gone. To persist data, mount storage from outside the container.

TypeWhat it isUse case
Named volumeManaged by Docker under /var/lib/docker/volumes.Preferred for databases & app data. Portable, backup-friendly.
Bind mountMaps a host directory into the container.Live source code in development.
tmpfsIn-memory, never written to disk.Sensitive temp data, scratch space.
bash
# Named volume (data survives container removal)
docker volume create pgdata
docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres:16

# Bind mount for live dev (host ./src -> container /app/src)
docker run -it -v "$(pwd)/src:/app/src" node:20-alpine sh

# Read-only mount + tmpfs
docker run -v config:/etc/app:ro --tmpfs /tmp myapp:1.0

# Volume management
docker volume ls
docker volume inspect pgdata
docker volume rm pgdata
docker volume prune            # remove unused volumes
⚠️ Back up your volumes Docker won't protect you from data loss. Back up named volumes regularly, e.g. by running a helper container that tars the volume into a file on the host: docker run --rm -v pgdata:/data -v $(pwd):/backup alpine tar czf /backup/pgdata.tgz -C /data .

9. Networking Intermediate

Docker provides several network drivers to control how containers communicate with each other and the outside world.

DriverBehaviorUse case
bridge (default)Private internal network on the host; containers get internal IPs.Single-host multi-container apps.
hostContainer shares the host's network stack (no isolation).Max performance, no port mapping needed (Linux).
noneNo networking at all.Fully isolated/batch jobs.
overlayMulti-host network across a Swarm cluster.Distributed services.
macvlanGives a container its own MAC/IP on the physical LAN.Legacy apps needing a real network identity.
💡 User-defined bridges give you DNS On the default bridge, containers can only reach each other by IP. On a user-defined bridge network, Docker provides automatic DNS — containers reach each other by name. Always create a custom network for multi-container apps.
bash
docker network create appnet
docker run -d --name db --network appnet postgres:16
docker run -d --name api --network appnet -e DB_HOST=db myapp:1.0
# 'api' can now reach the database simply at hostname: db

docker network ls
docker network inspect appnet
docker network connect appnet some-container
docker network prune

Port publishing

bash
docker run -p 8080:80 nginx        # host:8080 -> container:80
docker run -p 127.0.0.1:8080:80 nginx # bind to localhost only
docker run -P nginx                 # publish all EXPOSEd ports to random host ports

10. Configuration, Env Vars & Secrets Intermediate

Keep configuration out of your image so the same image works in every environment (12-factor app).

bash
# Individual env vars
docker run -e LOG_LEVEL=debug -e PORT=3000 myapp:1.0

# From an env file
docker run --env-file ./app.env myapp:1.0
🔴 Never bake secrets into images or ENV in the Dockerfile Anything in a Dockerfile layer (including ENV SECRET=... or COPY .env) is permanently visible via docker history to anyone with the image. For build-time secrets use BuildKit secret mounts; for run-time secrets use Docker secrets (Swarm), bind-mounted files, or an external manager (Vault, cloud secret stores).
dockerfile — BuildKit secret mount (not stored in image)
# syntax=docker/dockerfile:1
RUN --mount=type=secret,id=npmtoken \
    NPM_TOKEN=$(cat /run/secrets/npmtoken) npm ci
bash
docker build --secret id=npmtoken,src=$HOME/.npmtoken -t myapp:1.0 .

11. Registries: Push & Pull Intermediate

A registry stores and distributes images. Docker Hub is the default public one; production teams usually use private registries (GHCR, AWS ECR, Google AR, Azure ACR, Harbor).

bash
docker login                          # Docker Hub
docker login ghcr.io -u USERNAME       # other registry

# Tag for the target registry, then push
docker tag myapp:1.0 ghcr.io/acme/myapp:1.0
docker push ghcr.io/acme/myapp:1.0

docker pull ghcr.io/acme/myapp:1.0
docker logout

# Run a local private registry for testing
docker run -d -p 5000:5000 --name registry registry:2
docker tag myapp:1.0 localhost:5000/myapp:1.0
docker push localhost:5000/myapp:1.0
✅ Tagging strategy for production Tag every build with an immutable identifier — the git commit SHA (myapp:git-a1b2c3d) or a semantic version (myapp:1.4.2). Optionally also move a :stable tag. This makes every deployment traceable and rollback trivial.

12. Docker Compose Advanced

Real apps have multiple services (web, API, DB, cache). Docker Compose defines and runs multi-container apps from a single declarative YAML file.

yaml — compose.yaml
services:
  api:
    build: .                    # build from local Dockerfile
    image: myapp:1.0
    ports: ["3000:3000"]
    environment:
      DB_HOST: db
      REDIS_URL: redis://cache:6379
    depends_on:
      db: { condition: service_healthy }
    restart: unless-stopped

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes: ["pgdata:/var/lib/postgresql/data"]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      retries: 5

  cache:
    image: redis:7

volumes:
  pgdata:
bash
docker compose up -d              # build + start everything
docker compose ps                 # status of services
docker compose logs -f api        # follow one service's logs
docker compose exec api sh        # shell into a service
docker compose build             # rebuild images
docker compose up -d --scale api=3 # scale a service
docker compose stop
docker compose down              # stop + remove containers/networks
docker compose down -v           # also remove named volumes
💡 Compose for dev, orchestrator for prod Compose is fantastic for local development and small single-host deployments. For multi-host production with self-healing, rolling updates, and autoscaling, graduate to Kubernetes or Docker Swarm. You can use multiple compose files (-f compose.yaml -f compose.prod.yaml) and profiles for environment overrides.

13. BuildKit & Advanced Builds Advanced

BuildKit is Docker's modern build engine (default in current versions). It brings parallel stage builds, better caching, secret/SSH mounts, and multi-platform builds via docker buildx.

bash
# Cache mount — persist a package cache between builds
# in Dockerfile:
# RUN --mount=type=cache,target=/root/.npm npm ci

# Build for multiple architectures (amd64 + arm64) and push
docker buildx create --use
docker buildx build --platform linux/amd64,linux/arm64 \
  -t ghcr.io/acme/myapp:1.0 --push .

# Use a registry as a shared build cache (great for CI)
docker buildx build \
  --cache-to type=registry,ref=ghcr.io/acme/myapp:cache \
  --cache-from type=registry,ref=ghcr.io/acme/myapp:cache \
  -t ghcr.io/acme/myapp:1.0 --push .

14. Security Best Practices Expert

Containers share the host kernel, so a compromised container can be a serious risk. Harden aggressively for production.

PracticeWhy / How
Run as non-rootAdd a USER in the Dockerfile. Combine with --user and drop the root account inside.
Minimal base imagedistroless/alpine/scratch → fewer packages = smaller attack surface.
Pin versions & digestsReproducible, avoids surprise upstream changes.
Scan imagesdocker scout cves, Trivy, Grype — fail CI on critical CVEs.
Read-only filesystem--read-only + --tmpfs /tmp. App can't be tampered with.
Drop capabilities--cap-drop ALL --cap-add NET_BIND_SERVICE — only what's needed.
No privileged modeAvoid --privileged; use --security-opt no-new-privileges.
Limit resources--memory, --cpus, --pids-limit to prevent DoS.
Sign & verify imagesCosign / Docker Content Trust to ensure provenance.
Don't mount the Docker socketMounting /var/run/docker.sock = giving root on the host.
bash — hardened run
docker run -d \
  --read-only --tmpfs /tmp \
  --cap-drop ALL --cap-add NET_BIND_SERVICE \
  --security-opt no-new-privileges \
  --user 1000:1000 \
  --memory 512m --cpus 1 --pids-limit 200 \
  --restart unless-stopped \
  myapp:1.0

# Scan an image for vulnerabilities
docker scout cves myapp:1.0
trivy image myapp:1.0

15. Image Optimization Advanced

  • Multi-stage builds — ship only artifacts, not build tools.
  • Small base images — alpine/distroless/slim.
  • Order layers by change frequency — maximize cache hits.
  • Combine RUN steps & clean caches in the same layer.
  • Use .dockerignore — keep the context lean.
  • Avoid installing extras — --no-install-recommends, no editors/curl in final image.
  • One concern per image — don't bundle unrelated tools.
bash — inspect & clean
docker images                         # compare sizes
docker history --no-trunc myapp:1.0     # see what each layer costs
docker system df                       # disk usage breakdown

# Reclaim space — be careful, this deletes!
docker system prune                    # stopped containers, unused nets, dangling images
docker system prune -a --volumes       # aggressive: also unused images + volumes

16. Healthchecks, Logging & Restart Policies Expert

Healthchecks

A HEALTHCHECK lets Docker (and orchestrators) know whether your app is actually working — not just that the process is alive. Compose/Swarm/K8s use it to gate traffic and restarts.

bash
docker ps                  # STATUS column shows (healthy)/(unhealthy)
docker inspect --format '{{.State.Health.Status}}' api

Logging

Best practice: apps should log to stdout/stderr; Docker captures it via a logging driver. In production, ship logs to a central system (Loki, ELK, CloudWatch, Datadog) instead of letting them fill the disk.

bash
docker logs --tail 100 -f api
docker logs --since 10m api

# Cap log size so it can't fill the disk (json-file driver)
docker run --log-opt max-size=10m --log-opt max-file=3 myapp:1.0

# Ship to a remote syslog/fluentd
docker run --log-driver=fluentd --log-opt fluentd-address=...:24224 myapp:1.0
⚠️ Configure log rotation globally The default json-file driver has no size limit — logs can silently fill your disk and take down the host. Set defaults in /etc/docker/daemon.json: {"log-driver":"json-file","log-opts":{"max-size":"10m","max-file":"3"}}.

Restart policies & graceful shutdown

Use --restart unless-stopped (or always) so containers recover after crashes/reboots. Handle SIGTERM in your app to finish in-flight work; Docker waits 10s (configurable with --stop-timeout) before SIGKILL.

18. Docker in CI/CD Expert

Docker shines in pipelines: a build produces an immutable image that flows unchanged through test → staging → prod.

yaml — GitHub Actions example
name: build-and-push
on: { push: { branches: [main] } }
jobs:
  docker:
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write }
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v6
        with:
          push: true
          tags: ghcr.io/acme/myapp:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

A typical pipeline: build → scan (Trivy/Scout) → test → push immutable tag → deploy (update the tag in your orchestrator/GitOps repo).

19. Debugging & Troubleshooting Expert

bash
docker logs CONTAINER             # first stop for "why did it crash?"
docker inspect CONTAINER          # config, mounts, network, exit code
docker exec -it CONTAINER sh      # poke around inside
docker events                     # live stream of daemon events
docker stats                      # resource usage
docker diff CONTAINER             # filesystem changes since start
# Override entrypoint to debug a crashing image
docker run -it --entrypoint sh myapp:1.0
SymptomLikely cause & fix
Container exits immediatelyThe main process finished or crashed. Check docker logs; ensure a long-running foreground process (not a daemon that backgrounds itself).
port is already allocatedAnother process/container uses the host port. Change the mapping or stop the other one.
Can't connect between containersNot on the same user-defined network, or using IP instead of service name.
Changes don't appearStale cached layer or old image running. Rebuild (--no-cache) and recreate the container.
"no space left on device"Disk full of images/volumes/logs. Run docker system df then docker system prune; set log rotation.
Permission denied on volumeUID mismatch between host files and container user. Align USER/ownership or use named volumes.
Exit code 137OOM-killed — container hit its memory limit. Raise --memory or fix the leak.

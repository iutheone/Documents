1. What is Kubernetes? Beginner

Kubernetes (often abbreviated K8s — "K", eight letters, "s") is an open-source container orchestration platform. It automates the deployment, scaling, networking, healing, and management of containerized applications across a cluster of machines.

Originally designed by Google (based on their internal system Borg) and donated to the Cloud Native Computing Foundation (CNCF) in 2014, it has become the de-facto standard for running containers at scale.

The problem it solves

Containers (e.g. Docker) package an application with its dependencies. But running containers in production raises hard questions:

  • How do I run 500 containers across 20 servers?
  • What happens when a container or whole server crashes at 3 AM?
  • How do I roll out a new version with zero downtime — and roll back if it breaks?
  • How do containers find and talk to each other as they come and go?
  • How do I scale up on Black Friday and scale down afterwards automatically?

Kubernetes answers all of these. You declare the desired state ("I want 5 replicas of this app"), and Kubernetes continuously works to make reality match that state.

💡 The core idea: Declarative + Reconciliation You tell Kubernetes what you want (declarative), not how to do it (imperative). Controllers constantly compare the actual state to your desired state and take action to close the gap. This control loop is the heart of Kubernetes.

Key benefits

FeatureWhat it gives you
Self-healingRestarts failed containers, replaces and reschedules dead ones, kills unresponsive ones.
Horizontal scalingScale apps up/down manually or automatically based on CPU/memory/custom metrics.
Service discovery & load balancingStable DNS names and IPs for groups of containers, with built-in load balancing.
Automated rollouts/rollbacksProgressive deployment of new versions, automatic rollback on failure.
Storage orchestrationAutomatically mount local, cloud, or network storage.
Secret & config managementManage sensitive data and configuration without rebuilding images.
Batch executionRun batch jobs and CI workloads, replacing failed containers.
⚠️ What Kubernetes is NOT It is not a PaaS (it doesn't build your code or dictate languages), not a traditional monolithic deployment tool, and not magic. It won't fix a badly designed application — but it gives you powerful primitives to run good ones reliably.

2. Cluster Architecture Beginner

A Kubernetes cluster is a set of machines (physical or virtual) called nodes. There are two kinds of nodes:

  • Control plane nodes — the "brain" that makes global decisions and detects/responds to events.
  • Worker nodes — run your actual application workloads (Pods).

Control plane components

ComponentRole
kube-apiserverThe front door of the cluster. Exposes the Kubernetes REST API. Every command and every component talks through it. The only component that talks to etcd.
etcdConsistent, highly-available key-value store. The single source of truth — all cluster state lives here. Back this up!
kube-schedulerWatches for newly created Pods with no node assigned, and picks the best node for them (based on resources, affinity, taints, etc.).
kube-controller-managerRuns controllers: Node controller, ReplicaSet controller, Job controller, etc. Each is a reconciliation loop.
cloud-controller-managerIntegrates with the underlying cloud provider (load balancers, nodes, routes, storage).

Worker node components

ComponentRole
kubeletThe node agent. Ensures the containers described in PodSpecs are running and healthy. Reports status back to the API server.
kube-proxyMaintains network rules on each node, enabling Service networking and load balancing (via iptables/IPVS/eBPF).
Container runtimeSoftware that actually runs containers — e.g. containerd, CRI-O. (Docker support was removed in v1.24; containerd is now standard.)
💡 Golden rule Everything goes through the API server. kubectl, controllers, the scheduler, and kubelets all read and write cluster state via the API server. Nothing talks to etcd directly except the API server.

Inspecting resources

bash
kubectl get pods                          # list pods in current namespace

5. Pods — The Atom of Kubernetes Beginner

A Pod is the smallest deployable unit in Kubernetes. It wraps one (or rarely a few tightly-coupled) containers that:

  • Share the same network namespace (same IP & port space — they reach each other via localhost).
  • Can share storage volumes.
  • Are always scheduled together on the same node, live and die together.
⚠️ You rarely create Pods directly A bare Pod is not self-healing — if it dies, nothing recreates it. In practice you create Pods through higher-level controllers like Deployments, which manage Pods for you. Understanding Pods is foundational, but Deployments are what you deploy.

A minimal Pod

yaml — pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.27
      ports:
        - containerPort: 80
      resources:
        requests: { cpu: "100m", memory: "128Mi" }
        limits:   { cpu: "500m", memory: "256Mi" }
bash
kubectl apply -f pod.yaml
kubectl get pod nginx-pod -o wide
kubectl describe pod nginx-pod          # see events, status

# Interacting with a running Pod
kubectl logs nginx-pod                   # container logs
kubectl logs -f nginx-pod               # follow (stream)
kubectl logs nginx-pod --previous       # logs of crashed prev container
kubectl exec -it nginx-pod -- /bin/sh   # shell inside
kubectl port-forward nginx-pod 8080:80  # access locally
kubectl cp nginx-pod:/etc/nginx/nginx.conf ./nginx.conf  # copy files

Pod lifecycle & restart policy

A Pod moves through phases: Pending → Running → Succeeded/Failed. Common statuses you'll debug: ContainerCreating, CrashLoopBackOff, ImagePullBackOff, Pending (unschedulable), OOMKilled.

Multi-container patterns & init containers

Pods can run helper containers alongside the main one:

  • Sidecar — companion container (e.g. log shipper, service-mesh proxy).
  • Init containers — run to completion before app containers start (e.g. wait for DB, run migrations).
yaml
spec:
  initContainers:
    - name: wait-for-db
      image: busybox:1.36
      command: ['sh', '-c', 'until nc -z db 5432; do echo waiting; sleep 2; done']
  containers:
    - name: app
      image: my-app:1.0

Health checks: probes

Probes let Kubernetes know if your container is healthy. Critical for production.

ProbeQuestion it answersOn failure
livenessProbeIs the app alive (not deadlocked)?Container is restarted.
readinessProbeIs the app ready to serve traffic?Pod removed from Service endpoints (no restart).
startupProbeHas a slow-starting app finished booting?Holds off liveness/readiness until done.
yaml
containers:
  - name: app
    image: my-app:1.0
    livenessProbe:
      httpGet: { path: /healthz, port: 8080 }
      initialDelaySeconds: 10
      periodSeconds: 10
    readinessProbe:
      httpGet: { path: /ready, port: 8080 }
      periodSeconds: 5
    startupProbe:
      httpGet: { path: /healthz, port: 8080 }
      failureThreshold: 30
      periodSeconds: 10   # allows up to 300s to start

6. Labels, Selectors & Annotations Beginner

This is the glue of Kubernetes. Labels are key/value pairs attached to objects. Selectors query objects by their labels. This is how a Service finds its Pods, how a Deployment manages its Pods, etc.

bash
kubectl label pod nginx-pod env=prod         # add label
kubectl label pod nginx-pod env=dev --overwrite
kubectl label pod nginx-pod env-              # remove label

# Selectors
kubectl get pods -l env=prod
kubectl get pods -l 'env in (prod,staging)'
kubectl get pods -l 'env!=dev,app=nginx'

Annotations are also key/value but for non-identifying metadata (build info, tool configuration, descriptions). They cannot be used in selectors.

7. ReplicaSets & Deployments Intermediate

A ReplicaSet ensures a specified number of identical Pod replicas are running. A Deployment is a higher-level controller that manages ReplicaSets, adding rolling updates, rollbacks, and revision history. You almost always use Deployments for stateless apps.

Deployment ──manages──▶ ReplicaSet (rev 2) ──manages──▶ [Pod] [Pod] [Pod] │ └── keeps history ──▶ ReplicaSet (rev 1, scaled to 0) ── for rollback
yaml — deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  labels: { app: web }
spec:
  replicas: 3
  selector:
    matchLabels: { app: web }   # MUST match template labels
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }
  template:                       # the Pod template
    metadata:
      labels: { app: web }
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports: [{ containerPort: 80 }]
          resources:
            requests: { cpu: "100m", memory: "128Mi" }
            limits:   { cpu: "500m", memory: "256Mi" }

Managing Deployments

bash
kubectl apply -f deployment.yaml
kubectl get deploy,rs,pods               # see the hierarchy

# Scaling
kubectl scale deployment web --replicas=5

# Rolling update (change the image)
kubectl set image deployment/web web=nginx:1.28
kubectl rollout status deployment/web    # watch progress

# Rollout history & rollback
kubectl rollout history deployment/web
kubectl rollout undo deployment/web       # back to previous
kubectl rollout undo deployment/web --to-revision=2
kubectl rollout restart deployment/web    # restart all pods (e.g. reload config)
kubectl rollout pause deployment/web       # pause mid-rollout
kubectl rollout resume deployment/web

Deployment strategies

StrategyHow it worksTrade-off
RollingUpdate (default)Gradually replace old Pods with new ones (controlled by maxSurge/maxUnavailable).Zero downtime; both versions run briefly.
RecreateKill all old Pods, then create new ones.Downtime, but only one version at a time.
Blue/GreenRun new version in parallel, switch Service to it.Instant cutover; needs 2× resources. (Done via labels/services or tools like Argo Rollouts.)
CanarySend a small % of traffic to new version, increase gradually.Safe, gradual; needs traffic splitting (Ingress/mesh/Argo Rollouts).

8. Namespaces Intermediate

Namespaces provide a way to divide a cluster into virtual sub-clusters — useful for separating teams, environments (dev/staging), or projects. They scope names (two Pods named web can coexist in different namespaces) and are the boundary for quotas and RBAC.

bash
kubectl get namespaces
kubectl create namespace dev
kubectl get pods -n dev                   # target a namespace
kubectl apply -f app.yaml -n dev
kubectl delete namespace dev             # deletes everything inside!

Default namespaces: default (your stuff), kube-system (control plane components), kube-public, kube-node-lease.

💡 Cross-namespace DNS A Service is reachable at <service>.<namespace>.svc.cluster.local. Within the same namespace, just <service> works. Note: namespaces are not a strong security boundary by themselves — combine with NetworkPolicies and RBAC.

9. Services & Networking Intermediate

Pods are ephemeral — they get new IPs when recreated. A Service provides a stable virtual IP and DNS name that load-balances across a dynamic set of Pods (selected by labels).

TypeExposesUse case
ClusterIP (default)Internal-only virtual IPInternal communication between services.
NodePortA static port on every node (30000–32767)Basic external access / dev.
LoadBalancerCloud provider's external load balancerProduction external access (on cloud).
ExternalNameA CNAME to an external DNS nameAlias an external service.
Headless (clusterIP: None)No virtual IP — returns Pod IPs directlyStatefulSets, custom load balancing.
yaml — service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: ClusterIP
  selector:
    app: web                   # routes to Pods with this label
  ports:
    - port: 80                  # the Service port
      targetPort: 80            # the container port
      protocol: TCP
bash
kubectl apply -f service.yaml
kubectl get svc
kubectl get endpoints web              # which Pod IPs back the service

# Quick imperative expose
kubectl expose deployment web --port=80 --target-port=80 --type=NodePort

# Test internally
kubectl run tmp --image=busybox -it --rm -- wget -qO- web:80

How Service networking works

  • kube-proxy programs iptables/IPVS rules on each node so that traffic to the Service's ClusterIP is load-balanced to backing Pods.
  • CoreDNS provides cluster DNS — that's how web.default.svc.cluster.local resolves.
  • A CNI plugin (Calico, Cilium, Flannel, etc.) implements the actual Pod-to-Pod network so every Pod gets a routable IP.

10. Ingress & Gateway API Advanced

A Service of type LoadBalancer gives you one external IP per service — expensive and limited. Ingress lets you route HTTP/HTTPS traffic to many services through a single entry point, with host/path-based routing and TLS termination.

Ingress is just an API object — you need an Ingress Controller (NGINX, Traefik, HAProxy, or a cloud one) running in the cluster to fulfill it.

Internet │ ┌──────▼───────┐ │ Ingress Ctrl │ (single external LoadBalancer / IP) └──────┬───────┘ ┌───────────┼─────────────┐ │ shop.com/api shop.com/ blog.com/ ▼ ▼ ▼ [api-svc] [web-svc] [blog-svc]
yaml — ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: shop
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts: [shop.com]
      secretName: shop-tls
  rules:
    - host: shop.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service: { name: api-svc, port: { number: 80 } }
          - path: /
            pathType: Prefix
            backend:
              service: { name: web-svc, port: { number: 80 } }
💡 The future: Gateway API The Gateway API (gateway.networking.k8s.io) is the modern, more expressive successor to Ingress — with role-oriented resources (GatewayClass, Gateway, HTTPRoute) supporting advanced traffic splitting, header matching, and cross-namespace routing. New projects should evaluate it.

11. ConfigMaps & Secrets Intermediate

Separate configuration from code (a 12-factor principle). ConfigMaps hold non-sensitive config; Secrets hold sensitive data (passwords, tokens, keys).

bash
# Create ConfigMap from literals / files
kubectl create configmap app-config \
  --from-literal=LOG_LEVEL=info \
  --from-literal=APP_MODE=production \
  --from-file=config.json

# Create Secret
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password='S3cr3t!'

kubectl get configmap app-config -o yaml
kubectl get secret db-secret -o yaml

Consuming config in a Pod

yaml
spec:
  containers:
    - name: app
      image: my-app:1.0
      envFrom:
        - configMapRef: { name: app-config }   # all keys as env vars
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef: { name: db-secret, key: password }
      volumeMounts:
        - name: cfg
          mountPath: /etc/config        # mount as files
  volumes:
    - name: cfg
      configMap: { name: app-config }
🔴 Secrets are NOT encrypted by default Kubernetes Secrets are only base64-encoded in etcd, not encrypted. For production you MUST: (1) enable encryption at rest for etcd, (2) restrict access via RBAC, and (3) ideally use an external secret manager (HashiCorp Vault, AWS/GCP/Azure Secret Managers via the External Secrets Operator or Secrets Store CSI Driver). Never commit Secrets to Git in plaintext — use Sealed Secrets or SOPS.

12. Storage: Volumes, PV, PVC Advanced

Container filesystems are ephemeral — data is lost when a container restarts. Kubernetes provides volumes for persistence.

ConceptMeaning
emptyDirTemporary volume tied to the Pod's life; good for scratch/cache.
PersistentVolume (PV)A piece of real storage in the cluster (provisioned by admin or dynamically). Cluster-scoped.
PersistentVolumeClaim (PVC)A user's request for storage (size + access mode). Binds to a PV.
StorageClassDefines a "type" of storage and enables dynamic provisioning — PVs created on-demand when a PVC is made.
Pod ──uses──▶ PVC (claim: "I need 10Gi RWO") │ binds to ▼ PV (actual 10Gi disk) ◀──provisioned by── StorageClass (e.g. AWS gp3)

Access modes

  • ReadWriteOnce (RWO) — mounted read-write by a single node.
  • ReadOnlyMany (ROX) — read-only by many nodes.
  • ReadWriteMany (RWX) — read-write by many nodes (needs NFS/CephFS/etc.).
  • ReadWriteOncePod — read-write by a single Pod only.
yaml — pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: standard      # omit to use cluster default
  resources:
    requests: { storage: 10Gi }
---
apiVersion: v1
kind: Pod
metadata: { name: app }
spec:
  containers:
    - name: app
      image: postgres:16
      volumeMounts:
        - { name: data, mountPath: /var/lib/postgresql/data }
  volumes:
    - name: data
      persistentVolumeClaim: { claimName: data }
bash
kubectl get pv,pvc,storageclass
kubectl describe pvc data

13. Workload Controllers Advanced

Beyond Deployments, Kubernetes offers specialized controllers:

StatefulSet — for stateful apps

For databases, queues, and clustered apps needing stable identity and stable storage. Each Pod gets a sticky name (db-0, db-1...), stable DNS, and its own PVC. Pods are created/deleted in order.

yaml — statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata: { name: db }
spec:
  serviceName: db          # headless service for stable DNS
  replicas: 3
  selector: { matchLabels: { app: db } }
  template:
    metadata: { labels: { app: db } }
    spec:
      containers:
        - name: db
          image: postgres:16
          volumeMounts: [{ name: data, mountPath: /var/lib/postgresql/data }]
  volumeClaimTemplates:        # a PVC per Pod, auto-created
    - metadata: { name: data }
      spec:
        accessModes: [ReadWriteOnce]
        resources: { requests: { storage: 20Gi } }

DaemonSet — one Pod per node

Runs a copy of a Pod on every (or selected) node. Perfect for node-level agents: log collectors (Fluent Bit), monitoring (node-exporter), CNI plugins, storage daemons.

yaml
apiVersion: apps/v1
kind: DaemonSet
metadata: { name: node-exporter }
spec:
  selector: { matchLabels: { app: node-exporter } }
  template:
    metadata: { labels: { app: node-exporter } }
    spec:
      containers:
        - { name: node-exporter, image: prom/node-exporter }

Job & CronJob — batch work

A Job runs Pods to completion (e.g. a migration, a batch import). A CronJob runs Jobs on a schedule (like Unix cron).

yaml
apiVersion: batch/v1
kind: CronJob
metadata: { name: backup }
spec:
  schedule: "0 2 * * *"          # every day at 02:00
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      backoffLimit: 3
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: my-backup:1.0
              command: ["/backup.sh"]
bash
kubectl get jobs,cronjobs
kubectl create job --from=cronjob/backup manual-backup   # trigger now

14. Scheduling: Affinity, Taints, Tolerations Advanced

The scheduler decides which node a Pod runs on. You can influence this:

nodeSelector — simplest

yaml
spec:
  nodeSelector:
    disktype: ssd      # only schedule on nodes labeled disktype=ssd

Affinity & anti-affinity — expressive

yaml
spec:
  affinity:
    nodeAffinity:                       # place near node attributes
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - { key: zone, operator: In, values: [us-east-1a] }
    podAntiAffinity:                    # spread replicas across nodes
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector: { matchLabels: { app: web } }
            topologyKey: kubernetes.io/hostname

Taints & tolerations — repel Pods

A taint on a node repels Pods unless they have a matching toleration. Used to reserve nodes (e.g. GPU nodes, control-plane nodes).

bash
kubectl taint nodes node1 gpu=true:NoSchedule
kubectl taint nodes node1 gpu=true:NoSchedule-   # remove (trailing -)
yaml
spec:
  tolerations:
    - { key: gpu, operator: Equal, value: "true", effect: NoSchedule }

Topology spread & PodDisruptionBudget

Use topologySpreadConstraints to evenly spread Pods across zones/nodes, and a PodDisruptionBudget (PDB) to limit how many Pods can be down during voluntary disruptions (node drains, upgrades).

yaml — pdb.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata: { name: web-pdb }
spec:
  minAvailable: 2          # keep at least 2 Pods during drains
  selector: { matchLabels: { app: web } }

15. Resources, Limits & Autoscaling Advanced

Requests vs Limits

  • Request — the guaranteed amount the scheduler reserves. Used for placement decisions.
  • Limit — the hard cap. Exceeding CPU = throttled; exceeding memory = OOMKilled.

These determine the Pod's QoS class: Guaranteed (requests==limits), Burstable (requests<limits), BestEffort (none set). Under memory pressure, BestEffort Pods are evicted first.

⚠️ Always set requests/limits in production Without requests, the scheduler can't place Pods well and nodes get overcommitted. Without memory limits, one leaky Pod can take down a node. Set CPU requests (and usually no CPU limit, or generous ones, to avoid throttling) and always set memory requests == memory limits for predictability.

Horizontal Pod Autoscaler (HPA)

Automatically scales the number of Pods based on CPU/memory or custom metrics. Requires metrics-server.

yaml — hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: web }
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: { type: Utilization, averageUtilization: 70 }
bash
kubectl autoscale deployment web --cpu-percent=70 --min=2 --max=10
kubectl get hpa
kubectl top pods                # live CPU/mem (needs metrics-server)
kubectl top nodes

ResourceQuota & LimitRange

Per-namespace governance: ResourceQuota caps total resource consumption; LimitRange sets default/min/max per container.

yaml

16. Security: RBAC & ServiceAccounts Expert

RBAC (Role-Based Access Control) governs who can do what. Four objects:

ObjectScopePurpose
RoleNamespaceA set of allowed actions (verbs) on resources.
ClusterRoleClusterSame, but cluster-wide or for cluster-scoped resources.
RoleBindingNamespaceGrants a Role to users/groups/ServiceAccounts.
ClusterRoleBindingClusterGrants a ClusterRole cluster-wide.

ServiceAccounts are identities for Pods/processes (vs. users for humans). Every Pod runs as a ServiceAccount.

yaml — rbac.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: { namespace: dev, name: pod-reader }
rules:
  - apiGroups: [""]
    resources: [pods, pods/log]
    verbs: [get, list, watch]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: { namespace: dev, name: read-pods }
subjects:
  - { kind: ServiceAccount, name: ci-bot, namespace: dev }
roleRef:
  { kind: Role, name: pod-reader, apiGroup: rbac.authorization.k8s.io }
bash
# Can I do X? (test permissions)
kubectl auth can-i create deployments -n dev
kubectl auth can-i delete nodes --as=system:serviceaccount:dev:ci-bot
kubectl auth can-i --list -n dev
✅ Principle of least privilege Grant the minimum permissions needed. Avoid cluster-admin for apps. Disable auto-mounting of the SA token (automountServiceAccountToken: false) for Pods that don't call the API.

17. Network Policies Expert

By default, all Pods can talk to all Pods. NetworkPolicies let you firewall traffic at the Pod level (requires a CNI that supports them, e.g. Calico/Cilium).

yaml — default-deny + allow
# 1) Default deny all ingress in the namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: default-deny, namespace: prod }
spec:
  podSelector: {}
  policyTypes: [Ingress]
---
# 2) Allow only the frontend to reach the api on port 8080
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: api-allow-frontend, namespace: prod }
spec:
  podSelector: { matchLabels: { app: api } }
  policyTypes: [Ingress]
  ingress:
    - from:
        - podSelector: { matchLabels: { app: frontend } }
      ports:
        - { protocol: TCP, port: 8080 }

Pod Security & hardening

The built-in Pod Security Admission enforces three levels (privileged, baseline, restricted) per namespace via labels. Harden containers with a securityContext:

yaml
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 2000
    seccompProfile: { type: RuntimeDefault }
  containers:
    - name: app
      image: my-app:1.0
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities: { drop: ["ALL"] }
💡 Policy engines For org-wide rules ("no :latest images", "must have resource limits", "no privileged Pods"), use a policy engine like Kyverno or OPA/Gatekeeper as a validating admission controller.

18. Helm — The Package Manager Advanced

Writing dozens of YAML files per app, per environment, gets unwieldy. Helm packages Kubernetes manifests into reusable, templated, versioned charts.

bash
# Add a repo and install a chart
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo nginx
helm install my-release bitnami/nginx --version 15.0.0

# Override values
helm install my-release bitnami/nginx -f values.yaml \
  --set replicaCount=3 --namespace web --create-namespace

# Lifecycle
helm list -A
helm upgrade my-release bitnami/nginx -f values.yaml
helm rollback my-release 1
helm uninstall my-release
helm template ./mychart            # render locally (no install)
helm lint ./mychart

Chart structure

mychart/ ├── Chart.yaml # name, version, dependencies ├── values.yaml # default config values ├── templates/ # templated manifests │ ├── deployment.yaml # uses {{ .Values.image.tag }} │ ├── service.yaml │ └── _helpers.tpl # reusable template snippets └── charts/ # bundled sub-charts (dependencies)
✅ Helm vs Kustomize Kustomize (built into kubectl via -k) does template-free overlays — great for environment variants of your own manifests. Helm excels at packaging and distributing apps with rich templating. Many teams use both: Helm for third-party apps, Kustomize for in-house ones.

20. Debugging & Troubleshooting Expert

The universal debugging workflow

  1. kubectl get pods — what's the status?
  2. kubectl describe pod X — read the Events at the bottom (most issues are explained here).
  3. kubectl logs X (and --previous) — application errors.
  4. kubectl get events --sort-by=.lastTimestamp — cluster-level clues.
  5. kubectl exec -it X -- sh — poke around inside.

Common errors decoded

StatusLikely causeFix
ImagePullBackOff / ErrImagePullWrong image name/tag, private registry, no credentials.Check image name; add imagePullSecrets.
CrashLoopBackOffApp crashes on start (bad config, missing dep, failing liveness probe).kubectl logs --previous; check env/config & probes.
PendingNo node has enough resources, or unsatisfiable affinity/taints.kubectl describe pod → check scheduler message; add nodes / relax requests.
OOMKilledContainer exceeded its memory limit.Raise memory limit or fix the leak.
CreateContainerConfigErrorMissing ConfigMap/Secret referenced.Create the referenced object.
Service has no endpointsService selector doesn't match Pod labels, or Pods not Ready.Compare selector vs Pod labels; check readiness probe.

21. Running in Production Expert

This is what separates a demo from a real platform. Below is a battle-tested checklist.

High Availability

  • 3+ control-plane nodes across availability zones (etcd needs an odd quorum: 3 or 5).
  • Worker nodes spread across multiple AZs; use topologySpreadConstraints + podAntiAffinity so replicas don't all land in one zone.
  • Multiple replicas for every critical service + a PodDisruptionBudget.
  • Cluster Autoscaler / Karpenter to add capacity automatically.

etcd backup & disaster recovery

etcd is the single source of truth. Back it up regularly and test restores.

bash
# Snapshot etcd
ETCDCTL_API=3 etcdctl snapshot save snapshot.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

etcdctl snapshot status snapshot.db -w table
✅ Also back up your manifests & volumes Use Velero to back up cluster resources and PersistentVolumes to object storage, enabling full cluster restore and migration. With GitOps, your manifests are already safe in Git.

Cluster & node upgrades

  • Upgrade order: control plane first, then workers. Never skip minor versions (1.28 → 1.29 → 1.30).
  • Drain nodes safely before maintenance; PDBs protect availability.
  • Keep the kubelet within one minor version of the API server (version skew policy).
bash
kubectl cordon node1                # mark unschedulable
kubectl drain node1 --ignore-daemonsets --delete-emptydir-data
# ...do maintenance / upgrade...
kubectl uncordon node1              # allow scheduling again

GitOps — the production deployment model

Instead of running kubectl apply by hand, declare desired state in Git and let a controller continuously sync the cluster to match. Tools: Argo CD, Flux.

Developer ──git push──▶ Git repo (manifests/Helm/Kustomize) │ watches ▼ Argo CD / Flux ──syncs──▶ Kubernetes cluster (auto-heal drift, audit trail, easy rollback = git revert)
  • Git is the single source of truth; every change is reviewed (PR) and audited.
  • Rollback = git revert. Drift is auto-corrected.
  • Separate config repos per environment, or use overlays/branches.

CI/CD pipeline shape

  1. CI: build image → scan (Trivy) → test → push to registry with an immutable tag (git SHA, never :latest).
  2. CD: update the image tag in the Git manifests → Argo CD/Flux deploys → progressive rollout (canary/blue-green via Argo Rollouts) → automated health checks → auto-rollback on failure.

Production readiness checklist

AreaMust-haves
ReliabilityMultiple replicas, anti-affinity, PDBs, readiness/liveness/startup probes, graceful shutdown (terminationGracePeriodSeconds + SIGTERM handling + preStop hook).
ResourcesRequests & limits on every container; ResourceQuotas/LimitRanges per namespace; HPA + Cluster Autoscaler.
SecurityRBAC least-privilege, NetworkPolicies (default-deny), Pod Security (restricted), non-root + read-only FS, image scanning, signed images, secrets encrypted/externalized, regular patching.
ObservabilityMetrics + dashboards, centralized logs, distributed tracing, alerting on SLOs, audit logging enabled.
Operationsetcd + Velero backups (tested restores!), documented upgrade process, GitOps, runbooks, DR plan.
ImagesSmall, pinned versions, multi-stage builds, distroless/minimal base, vulnerability-scanned in CI.
NetworkingIngress/Gateway with TLS (cert-manager), rate limiting/WAF, internal vs external separation.
⚠️ Graceful shutdown matters When a Pod is terminated, Kubernetes sends SIGTERM, waits terminationGracePeriodSeconds (default 30s), then SIGKILL. Your app must catch SIGTERM, stop accepting new requests, and finish in-flight ones. A preStop hook with a small sleep helps the Service stop routing before the app dies, avoiding dropped requests during rollouts.

22. CRDs, Operators & the Ecosystem Expert

Custom Resource Definitions (CRDs) let you extend the Kubernetes API with your own resource types. An Operator is a custom controller that manages a CRD — encoding human operational knowledge ("how to back up and fail over this database") into software.

bash
kubectl get crds
kubectl api-resources --api-group=monitoring.coreos.com

Examples you'll meet in production: Prometheus Operator, cert-manager, the various database operators (CloudNativePG, Strimzi for Kafka), Argo CD, Istio/Linkerd (service mesh), Crossplane (infrastructure as Kubernetes resources).
